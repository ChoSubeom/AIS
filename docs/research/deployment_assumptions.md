# AIS 배포 가정 분석

> 회의적 연구 노트 — AIS가 실제로 배포되려면 무엇이 필요한가?

---

## 1. 핵심 질문

AIS 명세서는 "기존 애플리케이션 코드 수정 없이 적용 가능한 투명한 프록시"를 목표로 한다. 이 주장이 성립하려면 어떤 조건이 충족되어야 하는가? 그리고 현실적으로 그 조건들이 얼마나 달성 가능한가?

---

## 2. AIS가 암묵적으로 전제하는 배포 가정들

### 2.1 가정 1: Root CA에 대한 신뢰가 존재한다

AIS AI Certificate는 Ed25519 키 쌍으로 서명된다. 이 서명이 의미를 가지려면 다음이 필요하다:

- **신뢰할 수 있는 발행자(issuer)**가 존재해야 한다
- 클라이언트가 해당 발행자의 공개키를 신뢰해야 한다
- 발행자의 개인키가 안전하게 보관되어야 한다

**문제**: TLS의 경우 이미 수십 년에 걸쳐 구축된 Web PKI (Let's Encrypt, DigiCert 등)가 존재한다. AI Certificate의 경우 이런 인프라가 전혀 없다. 누가 AI 모델 인증서를 발행하는가? 모델 제조사가 자체 CA를 운영하는가? 독립적인 제3자 CA가 존재하는가?

MVP에서는 이 문제를 "로컬 신뢰 앵커"로 회피하고 있다. 이는 개발 환경에서는 작동하지만, 실제 배포에서는 핵심 문제가 된다. 인증서 체인이 없는 AI Certificate는 자기 서명 TLS 인증서와 동일한 신뢰 수준을 가진다 — 즉, 검증 가능한 신뢰가 없다.

### 2.2 가정 2: 개인키가 안전하게 보관된다

명세서는 다음을 명시한다:

> "클라이언트와 서버의 개인 키는 안전하게 보관된다."

이것은 위협 모델의 **전제조건**이지 AIS가 제공하는 보장이 아니다. 즉, AIS는 키 관리 문제를 해결하지 않는다 — 이미 안전한 키 관리가 존재한다는 것을 가정한다.

현실에서 개인키 보호는 다음 중 하나를 필요로 한다:
- HSM(Hardware Security Module)
- TEE 환경
- 클라우드 KMS (AWS KMS, Google Cloud KMS 등)
- 또는 기존의 신뢰할 수 있는 키 관리 인프라

AIS는 이 중 어느 것도 제공하지 않으며, 이를 사전 요구사항으로 취급한다. 결과적으로 AIS는 이미 안전한 환경에서만 추가적인 보안을 제공한다는 역설이 발생한다.

### 2.3 가정 3: 모델 가중치 파일에 직접 접근 가능하다

AI Certificate 검증은 다음을 요구한다:

```
verify_certificate(certificate, model_bytes)
```

즉, 클라이언트(또는 검증자)가 모델 가중치 파일의 전체 바이트에 접근할 수 있어야 한다.

**현실적 문제들**:

1. **70B+ 모델의 경우**: 검증 시 수십~수백 GB 파일의 SHA3-256 해시를 계산해야 한다. 이는 초기화 시 수십 초의 지연을 유발한다 (명세서는 "< 10s" 목표를 명시하지만, 70B 모델을 일반 디스크에서 읽는 데만 수십 초가 소요될 수 있다).

2. **API 기반 모델 서비스**: OpenAI, Anthropic 등의 클라우드 API를 통해 모델을 사용하는 경우, 클라이언트는 모델 가중치에 접근할 수 없다. AIS AI Certificate는 이 시나리오에서 작동하지 않는다.

3. **양자화/최적화 모델**: 동일한 모델이 GGUF, GPTQ, AWQ 등 다양한 형식으로 양자화될 때 가중치 해시가 달라진다. 각 변형에 대한 별도 인증서가 필요한가? 이는 인증서 관리 복잡도를 크게 증가시킨다.

### 2.4 가정 4: 프록시가 신뢰 경계 내에 있다

AIS 프록시는 다음을 수행한다:
- 세션 키를 보유
- 평문 요청을 처리
- 백엔드에 포워딩

이 아키텍처는 **프록시 자체가 신뢰 경계 내에 있어야 한다**는 전제가 필요하다. 만약 프록시가 손상(compromise)된다면:
- 모든 세션 키가 노출된다
- 모든 요청/응답이 평문으로 노출된다
- 감사 로그가 조작될 수 있다

이 문제는 TEE 없이는 소프트웨어 수준에서 해결할 수 없다. AIS MVP는 TEE를 명시적으로 배제했으므로, 프록시 자체의 무결성은 보장되지 않는다.

## 3. 채택 장벽 분석

### 3.1 기술적 채택 장벽

| 장벽 | 심각도 | 현재 AIS의 대응 |
|---|---|---|
| AI Certificate 발행 인프라 부재 | 높음 | "로컬 신뢰 앵커"로 회피 |
| 대용량 모델 파일 해시 성능 | 중간 | "< 10s" 목표, 미검증 |
| 클라우드 API 호환성 없음 | 높음 | 로컬 모델 전용으로 스코프 축소 |
| 키 관리 인프라 필요 | 높음 | 외부 의존성으로 처리 |
| 인증서 갱신/폐기 메커니즘 없음 | 중간 | MVP에서 명시적 제외 |

### 3.2 생태계 채택 장벽

더 심각한 문제는 기술적 장벽이 아닌 **생태계 채택** 문제다.

TLS가 성공한 이유는 단순히 좋은 기술이었기 때문이 아니다. 웹 브라우저 제조사, 서버 소프트웨어(Apache, Nginx), CA 업계가 동시에 채택했기 때문이다. AIS가 의미 있는 보안 보장을 제공하려면:

1. **모델 제조사(Anthropic, Meta, Mistral 등)** 가 AI Certificate를 발행해야 한다
2. **모델 런타임(vLLM, Ollama 등)** 이 AIS 프로토콜을 지원해야 한다
3. **클라이언트 프레임워크(LangChain, llamaindex 등)** 가 AIS 세션을 사용해야 한다
4. **독립적인 CA 생태계** 가 구축되어야 한다

이 중 어느 하나라도 빠지면 AIS는 자체 서명된 인증서처럼 의미 없는 보증만 제공한다. 현재 이 중 어느 것도 존재하지 않는다.

## 4. 가장 현실적인 사용 사례

솔직하게 평가했을 때, AIS가 현재 상태에서 실질적인 가치를 제공하는 시나리오는 매우 좁다:

### 사용 사례 A: 기업 내부 자체 호스팅 AI 인프라

- 기업이 자체 모델을 운영하는 경우
- 내부 CA를 직접 운영할 수 있는 경우
- 감사 로그가 규제 요구사항(HIPAA, GDPR 등)으로 필요한 경우
- 에이전트 도구 호출에 대한 세분화된 제어가 필요한 경우

이 시나리오에서는 AIS가 실질적인 가치를 제공할 수 있다.

### 사용 사례 B: AI 보안 연구 및 학술 목적

AIS가 제안하는 AI 특화 신뢰 모델 자체는 연구 가치가 있다. 특히:
- AI 인프라 신뢰 모델의 표준화 논의를 위한 참조 구현
- EU AI Act 대응을 위한 감사 메커니즘 연구

## 5. 명세서가 과소평가하는 문제

### 5.1 인증서 폐기(Revocation) 없는 PKI의 위험성

MVP에서 폐기를 제외했지만, 이는 운영 환경에서 치명적 문제가 될 수 있다. 모델이 취약점으로 인해 교체되어야 할 때, 기존 인증서를 폐기할 수 없다면 구 버전 모델이 계속 "유효한" 것으로 인식된다.

### 5.2 분산 환경에서의 감사 체인

현재 AIS 감사 체인은 인메모리 단일 프로세스다. 멀티-인스턴스 배포(로드 밸런서 뒤의 여러 프록시)에서 감사 체인은 분산 합의 없이 유지될 수 없다. 이 문제는 "MVP 이후" 로 미루어졌지만, 실제 운영 환경에서는 피할 수 없는 문제다.

### 5.3 백엔드 API 키 처리

현재 AIS 프록시는 클라이언트의 `Authorization` 헤더를 백엔드에 그대로 포워딩한다. 이는 프록시가 백엔드 API 키에 접근할 수 있음을 의미한다. AIS가 신뢰 레이어를 추가하면서, 동시에 새로운 신뢰 경계 문제를 도입하는 역설이 발생한다.

---

## 6. 결론

AIS의 배포 모델은 다음 조건 하에서만 현실적이다:

1. **폐쇄된 생태계**: 단일 조직이 CA, 모델, 런타임, 클라이언트를 모두 제어하는 환경
2. **로컬 인프라**: 클라우드 API를 사용하지 않고 온프레미스 모델을 운영하는 환경
3. **기술적 수용 가능**: AIS 프록시의 추가 레이턴시(~38µs 오버헤드)가 허용 가능한 환경

**개방된 생태계에서 AIS를 의미 있게 배포하려면** 현재 MVP에서 명시적으로 배제한 구성요소들(투명성 로그, 폐기 인프라, TEE 연동, 다중 CA 생태계)이 먼저 갖추어져야 한다. 이 구성요소들은 단순한 "미래 작업"이 아니라, AIS의 핵심 보안 보증이 성립하기 위한 전제 조건이다.

---

*이 분석은 AIS를 무효화하려는 것이 아니다. AIS 명세서 자체가 MVP의 한계를 솔직하게 명시하고 있으며, 이는 건전한 연구 접근 방식이다. 이 문서는 그 한계를 더 명확하게 드러내어 향후 연구 방향 설정에 기여하려는 목적이다.*
