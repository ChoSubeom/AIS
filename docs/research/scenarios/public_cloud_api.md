# 시나리오 평가: 퍼블릭 클라우드 AI API

> 회의적 시나리오 분석 — AIS는 OpenAI/Anthropic/Gemini API 환경에서 의미 있는 가치를 제공하는가?

---

## 시나리오 개요

**환경:**
- 소프트웨어 회사가 OpenAI GPT-4o, Anthropic Claude, Google Gemini API 중 하나를 사용한다
- 모델은 외부 클라우드 공급자가 운영하며 클라이언트는 API만 호출한다
- 클라이언트 환경: SaaS 애플리케이션 (웹/모바일), 백엔드 서비스

**배포 모델:**
```
애플리케이션 서버
    ↓ HTTPS (TLS 1.3)
OpenAI / Anthropic / Google API 엔드포인트
    ↓ (공급자 내부 인프라)
모델 실행 (공급자 관할)
```

**신뢰 경계:**
- 모델 서버는 클라이언트가 제어하지 않는다
- 모델 가중치에 접근할 수 없다
- 공급자의 인프라와 보안 관행을 신뢰해야 한다

**운영 제약:**
- API 키 기반 인증
- 공급자가 정의한 API 스펙을 따라야 한다
- 공급자가 정의한 레이트 리밋, SLA 준수 필요

---

## 위협 모델

**현실적으로 빈번한 위협:**

1. **API 키 탈취**: 키가 코드에 하드코딩되거나 로그에 노출
2. **프롬프트 인젝션**: 사용자 입력을 통한 시스템 프롬프트 우회 또는 데이터 추출
3. **과도한 권한**: 애플리케이션이 필요 이상의 API 권한 보유

**이론적이지만 낮은 빈도:**

4. **MITM 공격**: 애플리케이션과 API 엔드포인트 사이 트래픽 가로채기
5. **재전송 공격**: 캡처된 API 요청 반복 제출

**이 환경에서 AIS와 관계없는 위협:**

6. 공급자 서버 내부 침해 (클라이언트 제어 밖)
7. 모델 학습 데이터 오염 (클라이언트 제어 밖)
8. 공급자의 모델 교체 (클라이언트가 막을 수 없음)

---

## 기존 보안 커버리지

**TLS 1.3:**
- OpenAI, Anthropic, Google 모두 HTTPS를 강제한다 ✅
- 전송 암호화: ✅
- 서버 인증 (공급자 도메인 인증): ✅
- MITM 방어 (CA 기반 서버 인증서 검증): ✅
- 재전송 방어 (TLS 레코드 시퀀스): ✅

**API 키 / OAuth:**
- Bearer Token 인증: ✅
- 키 로테이션 지원: ✅
- 조직/프로젝트 기반 접근 제어 (OpenAI 조직, Anthropic 워크스페이스): ✅

**공급자 제공 감사 로그:**
- OpenAI Usage Dashboard: 요청 수, 토큰 수, 모델별 사용량 ✅
- Anthropic Console: 유사한 사용량 추적 ✅
- 공급자 SIEM 연동 (enterprise 플랜): 일부 가능 ✅

**애플리케이션 레이어 로깅:**
- 애플리케이션이 요청/응답을 자체 로그에 기록: 이미 표준 관행 ✅

**AIS 없이 이미 해결된 것들:**
이 환경에서 가장 현실적인 보안 위협(API 키 탈취, 프롬프트 인젝션)은 AIS가 다루지 않는 영역이다. 전송 보안은 TLS가 이미 강력하게 제공한다.

---

## AIS가 추가하는 것 (있다면)

각 AIS 구성요소를 이 시나리오에서 비판적으로 평가한다.

### AI Certificate / 모델 신원

**주장**: 실행 중인 모델이 특정 가중치와 동일한지 검증한다.

**실제 평가**:

이 주장은 이 시나리오에서 **원천적으로 불가능하다**.

AIS AI Certificate 검증의 전제는 다음과 같다:
```
verify_certificate(certificate, model_bytes)
```

클라이언트는 OpenAI의 GPT-4o 가중치에 접근할 수 없다. Anthropic도, Google도 모델 가중치를 공개하지 않는다. 검증에 필요한 `model_bytes` 자체가 존재하지 않는다.

또한: 공급자가 모델을 업데이트해도 API 버전이 동일하다면 클라이언트는 알 수 없다. 이것은 AIS가 아닌 공급자 정책의 문제다.

**결론**: AI Certificate 기능은 이 환경에서 **완전히 적용 불가능하다**.

### 인증된 추론 세션

**주장**: 각 추론 요청이 AIS 세션에 귀속되어 무결성이 보호된다.

**실제 평가**:

AIS 세션은 클라이언트와 AIS 프록시 사이에 위치한다. AIS 프록시는 요청을 받아 OpenAI API로 포워딩한다. 즉:

```
클라이언트 →[AIS 세션]→ AIS 프록시 →[HTTPS+TLS]→ OpenAI API
```

이 구성에서 AIS 세션이 추가하는 것:
- 클라이언트와 프록시 사이의 MAC 검증 (TLS로 이미 보호된 구간)
- 시퀀스 카운터 (TLS 레코드 레이어로 이미 보호된 구간)

**이것은 TLS 위에 TLS와 기능적으로 유사한 레이어를 중복 추가하는 것이다** (`tls_overlap.md` §2.2).

**결론**: 인증된 세션 기능은 이 환경에서 TLS와 중복이다. 추가되는 보안 가치가 없다.

### 재전송 저항

**주장**: 동일 추론 요청의 재전송을 AIS 레이어에서 탐지하고 거부한다.

**실제 평가**:

이 환경에서 재전송 공격의 현실적 위험을 평가한다:

- TLS 레코드 레이어가 이미 전송 수준 재전송을 방어한다
- OpenAI API 자체가 요청마다 고유한 응답 ID를 반환한다
- 공격자가 API 키 없이 재전송을 시도해도 인증 실패한다
- 공격자가 API 키를 가지고 있다면, 재전송보다 새로운 요청을 직접 보내는 것이 더 쉽다

AIS 재전송 탐지의 유일한 의미 있는 케이스: 같은 세션 내에서 클라이언트 버그로 동일 요청이 반복되는 경우. 이것은 AIS 없이 애플리케이션 레이어에서 처리할 수 있다.

**결론**: 재전송 저항은 이 환경에서 의미 있는 추가 보안을 제공하지 않는다.

### 감사 체인

**주장**: 추론 요청의 변조 불가 감사 로그를 생성한다.

**실제 평가**:

이것이 이 환경에서 AIS의 가장 약한 주장이다.

공급자 자체 감사:
- OpenAI, Anthropic, Google 모두 요청 로그를 기록한다
- Enterprise 플랜에서는 더 세부적인 감사 데이터를 제공한다

애플리케이션 자체 로깅:
- 모든 성숙한 애플리케이션은 API 요청을 자체적으로 로깅한다
- OpenTelemetry + SIEM으로 더 풍부한 감사 데이터를 구성할 수 있다

AIS 감사 체인의 추가 가치:
- 공급자 로그가 조작되는 시나리오? → 공급자를 신뢰하지 않는다면 그 API를 애초에 사용하면 안 된다
- 애플리케이션 자체 로그가 조작되는 시나리오? → AIS 프록시가 동일 인프라에 있다면 같이 조작될 수 있다

**결론**: 감사 체인 기능은 이 환경에서 추가 가치가 없다. 기존 로깅 인프라로 충분하거나 더 낫다.

### Capability 모델

**주장**: API 호출에 대한 세분화된 권한 제어를 제공한다.

**실제 평가**:

공급자 API는 이미 자체 접근 제어를 제공한다:
- OpenAI: 조직, 프로젝트, API 키별 권한
- 모델별 접근 제어 가능

애플리케이션 레이어의 Gateway 패턴(API Gateway, 사이드카 프록시)으로 도구 호출을 제한하는 것이 AIS-Capability보다 더 범용적이다.

**결론**: Capability 모델도 이 환경에서 기존 대안으로 충분히 구현 가능하다.

## 더 강한 대안

이 환경에서 AIS보다 강하거나 더 직접적인 대안들:

**비밀 관리 시스템 (HashiCorp Vault, AWS Secrets Manager):**

API 키 탈취(이 환경의 가장 현실적인 위협)를 방어한다. AIS는 이를 다루지 않는다.

**WAF / API Gateway:**

프롬프트 인젝션(이 환경의 두 번째로 현실적인 위협)을 부분적으로 완화한다. AIS는 이를 다루지 않는다.

**공급자 제공 엔터프라이즈 기능:**

OpenAI, Anthropic의 엔터프라이즈 플랜은 더 세부적인 감사, 접근 제어, 데이터 격리를 제공한다.

**OpenTelemetry + SIEM:**

이 환경에서 관찰 가능성과 감사는 AIS보다 표준 도구로 더 잘 달성된다.

## 채택 장벽

**기술적 불가능성:**

AI Certificate 검증의 전제(모델 가중치 접근)가 충족될 수 없다. 이것은 기술적 장벽이 아니라 구조적 불가능성이다.

**추가 레이어의 운영 비용:**

AIS 프록시를 운영하면서 공급자 API 중단, 레이턴시 증가, 추가 복잡도를 감수해야 한다. 추가되는 보안 가치가 없다면 이 비용은 정당화되지 않는다.

**공급자 API 변경에 대한 취약성:**

공급자가 API 스펙을 변경하면 AIS 프록시도 업데이트해야 한다. 유지 관리 부담이 추가된다.

---

## 정직한 결론

**판정: AIS 불필요**

이 시나리오에서 AIS의 핵심 기능들은 다음 중 하나다:
- 원천적으로 적용 불가능 (AI Certificate, 모델 가중치 접근 불가)
- TLS와 중복 (세션 MAC, 재전송 탐지)
- 기존 인프라로 더 잘 달성됨 (감사, Capability)

AIS를 이 환경에 적용하는 것은 아무런 보안 개선 없이 운영 복잡도와 레이턴시만 추가하는 결과가 된다.

이 환경의 실제 위협(API 키 탈취, 프롬프트 인젝션)은 AIS의 설계 범위 밖이며, 다른 도구로 더 직접적으로 다루어야 한다.

**이 결론은 AIS에 대한 비판이 아니다.** AIS가 다른 환경(on-prem, 고보증 내부 인프라)에서 가치를 가질 수 있더라도, 이 환경에서는 불필요하다. 솔직한 scope 인식이 중요하다.

## 신뢰 수준

**신뢰도: 높음 (High)**

이 결론에 대한 신뢰도는 높다. 주요 이유:

1. AI Certificate 불가능성은 구조적 사실이다 — 클라이언트가 모델 가중치에 접근할 수 없다는 것은 논쟁의 여지가 없다
2. TLS와의 중복은 이미 `tls_overlap.md`에서 상세히 분석되었다
3. 공급자 감사 기능의 존재는 검증 가능한 사실이다

불확실성이 남아 있는 부분:
- 미래에 공급자들이 AI Certificate 발행을 시작한다면 상황이 바뀔 수 있다 (현재로서는 투기적)
- 특수한 컴플라이언스 요구사항이 있는 경우 예외적 상황이 있을 수 있다
