# 시나리오 평가: 기업 내부 AI 에이전트

> 회의적 시나리오 분석 — AIS는 사내 자동화 에이전트 환경에서 의미 있는 가치를 제공하는가?

---

## 시나리오 개요

**환경:**
- 중대형 소프트웨어 회사가 내부 AI 에이전트를 운영한다
- 에이전트는 자율적으로 도구를 호출하고 액션을 수행한다
- 에이전트 실행 환경: Kubernetes 클러스터 (온프레미스 또는 프라이빗 클라우드)
- 에이전트 LLM: 자체 호스팅 오픈소스 모델 (Mistral, CodeLlama 등)

**에이전트가 접근하는 도구:**
```
AI 에이전트
    ├── Jira API       (티켓 생성, 수정, 마감)
    ├── Slack API      (메시지 전송, 채널 알림)
    ├── PostgreSQL DB  (데이터 조회, 일부 쓰기)
    ├── GitHub API     (PR 생성, 코드 리뷰 요청)
    └── CI/CD          (배포 트리거)
```

**신뢰 경계:**
- 회사가 에이전트 런타임, LLM 서버, 도구 API를 모두 제어한다
- 내부 IAM (Okta, Active Directory 등) 이 도구 접근을 관리한다
- 에이전트 요청은 job queue (Kafka, RabbitMQ 등)를 통해 처리된다

**운영 제약:**
- 에이전트는 비동기로 실행된다 — job queue에서 꺼낸 작업을 처리
- 일부 작업은 재시도(retry) 로직이 있다 (네트워크 실패 시 동일 작업 재실행)
- 에이전트 실수가 비즈니스 영향을 준다 (잘못된 Jira 티켓, 잘못된 배포)
- 감사 로그가 내부 정책 또는 SOC 2 요구사항으로 필요하다

---

## 위협 모델

**현실적으로 빈번한 위협:**

1. **중복 실행 (Duplicate Execution)**: Job queue의 at-least-once 전달로 동일 작업이 두 번 실행 → 티켓 중복 생성, 배포 두 번 트리거, DB 데이터 중복 삽입
2. **에이전트 로그 불투명성**: 에이전트가 무엇을 했는지 사후에 정확히 알기 어려움
3. **의도하지 않은 도구 호출**: 에이전트가 지시받지 않은 도구를 호출 (예: Slack 메시지 불필요하게 전송)
4. **자격증명 탈취 후 에이전트 오용**: 에이전트 서비스 계정 탈취

**이론적이지만 낮은 빈도:**

5. **프롬프트 인젝션을 통한 도구 오남용**: 외부 데이터(GitHub 이슈, DB 레코드)에 포함된 악의적 명령으로 에이전트 유도 (AIS 범위 밖)

**이 환경에서 범위 밖:**

6. 외부 인터넷 공격자 (내부 에이전트는 외부에서 직접 접근 불가)
7. 모델 학습 단계 백도어

---

## 기존 보안 커버리지

**TLS / mTLS:**
- 에이전트와 도구 API 간 전송 암호화: ✅
- 서비스 계정 기반 mTLS (Kubernetes의 경우 Istio/Envoy): ✅
- 재전송 방어 (TLS 레코드 레이어): ✅

**OAuth / 서비스 계정:**
- 도구별 OAuth 토큰으로 에이전트 신원 인증: ✅
- 스코프 기반 권한 제어 (Jira API 스코프, GitHub 앱 권한): ✅
- 토큰 만료 및 갱신: ✅

**Job Queue / 멱등성 처리:**
- Kafka의 exactly-once semantics (트랜잭션 설정 시): ✅
- 애플리케이션 레이어 idempotency key: 구현 가능 ✅

**기존 감사 시스템:**
- Jira, Slack, GitHub 자체 감사 로그: ✅
- Kubernetes audit log: ✅
- 데이터베이스 audit log (PostgreSQL pg_audit 등): ✅

**AIS 없이 이미 해결된 것들:**
- 도구별 접근 권한: OAuth 스코프로 해결
- 전송 보안: TLS로 해결
- 중복 실행 방지: 멱등성 처리로 해결 (AIS와 무관)
- 도구별 감사: 각 도구의 자체 감사로 해결

**AIS가 추가하지 않는 것:**
OAuth 토큰으로 이미 "누가 어떤 도구에 접근했는가"는 관리된다.
Kubernetes + Istio가 이미 서비스 간 mTLS와 정책을 제공한다.

## AIS가 추가하는 것 (있다면)

### AI Certificate / 모델 신원

**주장**: 에이전트가 사용하는 LLM이 승인된 모델인지 검증한다.

**실제 평가**:

이 시나리오에서 "모델이 교체되었는가?"라는 질문은 현실적이다. 에이전트가 의도하지 않게 다른 모델 버전(파인튜닝된 버전, 오래된 버전)을 사용한다면 결과가 달라질 수 있다.

**그러나:**
- Kubernetes에서 컨테이너 이미지 digest 고정이 이미 모델 버전 고정을 보장한다
- GitOps / IaC 파이프라인이 배포된 모델을 추적한다
- AIS AI Certificate는 이 환경에서 이미 있는 인프라의 중복이 될 가능성이 높다

**결론**: 모델 신원 기능은 이 환경에서 기존 Kubernetes 인프라로 대체 가능하다.

### 재전송 저항

**주장**: 동일 추론 요청의 재전송을 탐지하고 거부한다.

**실제 평가**:

이것이 이 시나리오에서 AIS의 가장 흥미로운 주장이다. 그러나 정확한 검토가 필요하다.

**Job queue 재전송과 AIS 재전송의 차이:**

Job queue에서 발생하는 중복 실행은 AIS 세션 수준에서의 재전송이 아니다. 동일한 작업이 **새로운 세션**으로 다시 실행된다. AIS의 시퀀스 카운터는 세션 내 재전송만 탐지하며, 새 세션으로 동일 작업을 재실행하는 것은 탐지하지 못한다 (`threat_model_limits.md` §2, TC-04).

즉, job queue 재전송 문제는 애플리케이션 레이어 멱등성 처리(idempotency key)로 해결해야 하며, AIS가 이를 다루지 않는다.

**AIS 재전송 탐지가 의미 있는 세부 케이스:**

동일 세션 내에서 동일한 시퀀스의 요청이 재전송되는 경우는 다음과 같다:
- 네트워크 레이어 패킷 재전송 (TLS가 이미 처리)
- 클라이언트 버그로 인한 동일 요청 반복 제출

이 케이스들은 TLS 레코드 레이어와 애플리케이션 레이어 멱등성 처리의 조합으로 이미 다루어진다.

**결론**: AIS 재전송 방어는 이 시나리오의 실제 문제(job queue 중복 실행)를 해결하지 못한다.

### 감사 체인

**주장**: 에이전트의 각 추론 요청에 대한 변조 불가 감사 로그를 생성한다.

**실제 평가**:

이것이 이 시나리오에서 AIS의 두 번째로 강한 주장이다.

기존 감사 시스템이 다루는 것:
- Jira: "에이전트 서비스 계정이 티켓 #1234를 생성했다" ✅
- GitHub: "에이전트가 PR #567을 열었다" ✅
- DB: "에이전트가 테이블 X를 쿼리했다" ✅

기존 감사 시스템이 다루지 않는 것:
- "에이전트가 어떤 프롬프트를 LLM에게 보냈는가?" → AIS 감사 체인이 기록
- "어떤 LLM 응답에 기반해 이 액션을 결정했는가?" → AIS가 부분적으로 연결 가능

**그러나:**
- AIS 감사는 요청 본문의 해시만 기록한다. 원본 프롬프트 내용은 기록하지 않는다
- "어떤 프롬프트 → 어떤 응답 → 어떤 도구 호출"의 인과 체인은 AIS 단독으로 구성되지 않는다
- SIEM + LLM 애플리케이션 자체 로깅이 더 풍부한 감사 데이터를 제공할 수 있다

**결론**: 에이전트 추론 감사 측면에서 AIS는 부분적인 기여를 할 수 있다. 그러나 완전한 에이전트 감사 체인을 구성하려면 AIS만으로 불충분하다.

### Capability 모델

**주장**: 에이전트가 어떤 도구를 호출할 수 있는지 명시적으로 제어한다.

**실제 평가**:

이것이 이 시나리오에서 AIS-Capability의 가장 직접적인 사용 사례다.

현재 OAuth 스코프의 한계:
- OAuth 스코프는 "이 계정이 Jira에 접근할 수 있다"를 제어한다
- "AI 에이전트가 배포 트리거를 호출할 수 있는가?"는 별도 정책이 없다면 서비스 계정 권한에 따라 결정된다
- 에이전트가 스스로 판단해 도구를 선택하는 자율성 자체를 제어하는 메커니즘이 없다

AIS-Capability가 제공할 수 있는 것:
- "이 세션에서 에이전트는 DB 읽기만 가능하고, 배포 트리거는 불가"
- Capability 토큰에 시간 제한, 호출 횟수 제한 포함 가능

**그러나:**
- AIS-Capability는 MVP에 포함되지 않는다. 현재 구현되지 않았다
- OPA(Open Policy Agent) + Envoy 조합이 유사한 도구 호출 정책을 구현할 수 있다
- Kubernetes의 RBAC + Admission Controller가 서비스 수준에서 접근을 제어한다

**결론**: Capability 개념 자체는 이 환경에서 의미 있지만, 현재 AIS MVP에는 구현되지 않았다. 유사한 기능은 기존 정책 엔진으로 이미 구현 가능하다.

## 더 강한 대안

**Kubernetes + Istio (서비스 메시):**

서비스 간 mTLS, 트래픽 정책, 감사 로그를 제공한다. Kubernetes 환경에서 AIS 프록시 레이어보다 더 성숙하고 광범위한 보안을 제공한다.

**OPA + Rego (정책 엔진):**

에이전트 도구 호출 정책을 세분화할 수 있다. AIS-Capability의 기능 집합과 상당히 겹친다. 이미 프로덕션에서 널리 사용된다.

**Kafka 트랜잭션 (exactly-once semantics):**

Job queue 중복 실행 문제는 AIS가 아닌 Kafka 트랜잭션으로 해결해야 한다.

**OpenTelemetry + SIEM:**

에이전트 추론 요청, 도구 호출, 응답을 OpenTelemetry로 수집하고 SIEM에서 분석하면 AIS 감사 체인보다 더 풍부한 가시성을 제공한다.

---

## 채택 장벽

**Kubernetes 환경에서의 이중 레이어:**

Istio mTLS가 이미 있는 환경에서 AIS 프록시를 추가하면 레이어 중복이 발생한다. 운영 복잡도가 증가하면서 추가 가치가 불분명하다.

**에이전트 프레임워크와의 호환성:**

LangChain, AutoGen, CrewAI 등 에이전트 프레임워크는 자체적인 도구 호출 메커니즘을 가진다. AIS 프록시와의 통합은 비표준 추가 작업이다.

**Job Queue 통합의 복잡성:**

비동기 에이전트 실행에서 AIS 세션 생명주기가 job의 생명주기와 어떻게 대응하는지 불명확하다. 단일 job이 여러 추론 요청을 할 경우 세션 관리가 복잡해진다.

## 정직한 결론

**판정: AIS 부분적으로 유용할 가능성 있음 — 특히 Capability 미구현으로 현재는 제한적**

이 시나리오는 AIS-Capability가 구현된다면 의미 있는 가치를 제공할 수 있는 환경이다. 에이전트 도구 호출 경계, 세분화된 권한 제어, 에이전트 추론 감사는 현재 기존 시스템에서 완전히 다루어지지 않는다.

**그러나 현재 MVP 상태에서:**
- AIS-Capability 미구현 → 가장 강한 가치 주장이 사용 불가
- 재전송 방어는 job queue 중복 실행 문제를 해결하지 못함
- 감사 체인은 에이전트 감사 전체를 커버하지 못함
- Kubernetes + Istio가 이미 더 성숙한 보안을 제공하는 경우 AIS의 추가 가치가 불분명

**핵심 판단**: 이 시나리오에서 AIS의 진짜 가치는 AIS-Capability에 달려 있다. 그것 없이는 "의외로 강할 가능성"보다 "기존 인프라로 충분"에 가깝다.

---

## 신뢰 수준

**신뢰도: 낮음 (Low)**

- AIS-Capability가 구현되지 않아 핵심 주장을 실제로 검증할 수 없다
- 에이전트 아키텍처는 환경마다 편차가 크다 — 단일 시나리오로 일반화하기 어렵다
- Job queue 재전송과 AIS 재전송의 관계에 대한 심층 분석이 필요하다
- 이 시나리오의 결론은 AIS-Capability 구현 여부에 따라 크게 달라질 수 있다
