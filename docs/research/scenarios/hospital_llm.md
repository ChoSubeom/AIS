# 시나리오 평가: 병원 내부 LLM

> 회의적 시나리오 분석 — AIS는 병원 내부 AI 환경에서 의미 있는 가치를 제공하는가?

---

## 시나리오 개요

**환경:**
- 중규모 병원이 온프레미스 LLM을 운영한다 (예: Llama-3 기반 파인튜닝 모델)
- 모델 서버는 병원 내부 네트워크에 위치하며 인터넷과 격리된다
- 사용자: 의사, 간호사, 행정 직원 (약 500명)
- 용도: 임상 문서 요약, 진단 보조, 진료 기록 검색

**배포 모델:**
```
병원 클라이언트 (의사/간호사 단말)
    ↓ HTTPS (내부 PKI)
AIS Proxy (온프레미스)
    ↓
LLM 서버 (온프레미스)
    ↓
EMR 시스템 (Epic, Cerner 등)
```

**신뢰 경계:**
- 병원 IT 부서가 내부 PKI를 운영한다
- 모델 서버는 병원이 직접 관리한다
- 단말기는 병원 MDM(Mobile Device Management)에 등록된다

**운영 제약:**
- HIPAA 감사 요구사항: 누가 어떤 환자 정보에 언제 접근했는가를 기록해야 한다
- 모델은 6개월마다 재학습·교체된다
- 24/7 운영, 다운타임 허용 매우 제한적
- IT 인력이 제한적이다 (전담 보안 엔지니어 1~2명)

---

## 위협 모델

**현실적으로 빈번한 위협:**

1. **내부자 위협**: 직원이 권한 없이 환자 기록에 접근하기 위해 LLM을 사용
2. **자격증명 탈취**: 피싱으로 의사 계정 탈취 후 LLM을 통해 환자 정보 추출
3. **감사 로그 조작**: 부적절한 접근 사실을 은폐하기 위해 로그 수정 시도
4. **모델 교체**: IT 관리자가 승인 없이 모델을 교체하거나 파인튜닝 데이터를 변조

**이론적이지만 낮은 빈도의 위협:**

5. **네트워크 내부 MITM**: 내부 네트워크 패킷 스니핑
6. **재전송 공격**: 캡처된 진단 요청을 반복 제출

**이 환경에서 범위 밖인 위협:**

7. 외부 인터넷 공격자 (네트워크 격리로 해당 없음)
8. 모델 학습 단계 백도어 (AIS 범위 밖)
9. 프롬프트 인젝션 (AIS-Semantic 범위, MVP에서 제외)

---

## 기존 보안 커버리지

병원이 이미 갖추고 있는 보안 레이어를 냉정하게 평가한다.

**TLS / 내부 PKI:**
- 내부 CA를 통한 서버 인증: ✅
- 전송 암호화: ✅
- 클라이언트 인증 (mTLS): 추가 구성 시 ✅
- 재전송 방어 (TLS 레코드 레벨): ✅

**OAuth / SAML / 기존 IAM:**
- 사용자 인증 (Active Directory, Azure AD): ✅
- 역할 기반 접근 제어 (RBAC): ✅
- LLM API 접근 권한 제어: OAuth 토큰으로 가능 ✅

**EMR 시스템 자체 감사:**
- HIPAA 준수를 위한 EMR 자체 접근 로그: ✅ (Epic, Cerner 등은 내장)
- 누가 어떤 환자 기록을 열람했는가: ✅

**AIS 없이 이미 해결된 것들:**
- 전송 보안: TLS로 해결
- 사용자 인증: IAM으로 해결
- 환자 기록 접근 감사: EMR 자체 감사로 해결
- RBAC: 기존 인프라로 해결

**AIS가 추가하지 않는 것:**
기존 인프라가 갖추어진 병원에서 TLS + mTLS + OAuth 조합으로
위의 대부분이 이미 다루어진다.
AIS의 전송 레이어 보안 주장은 이 환경에서 중복이다.

---

## AIS가 추가하는 것 (있다면)

각 AIS 구성요소를 비판적으로 평가한다.

### AI Certificate / 모델 신원

**주장**: 현재 실행 중인 LLM이 승인된 가중치와 동일한지 검증한다.

**실제 평가**:

병원 IT가 "모델이 교체되었는가?"를 묻는 시나리오는 현실적이다.
IT 관리자가 악의적으로 또는 실수로 다른 모델을 배포할 수 있다.
TLS 서버 인증서는 서버 *프로세스*는 인증하지만, 어떤 모델 파일이 로드되었는지는 인증하지 않는다.

**그러나:**
- Docker Image Digest + signed container가 동일한 보증을 더 강하게 제공한다
- 병원 IT가 이미 소프트웨어 배포에 SCCM, Ansible 등의 변경 관리 시스템을 사용한다면 모델 파일 무결성은 그 레이어에서 관리 가능하다
- AIS AI Certificate는 TEE 없이는 소프트웨어 수준 근사치다 (`tls_overlap.md` §3.1)

**결론**: 모델 신원 기능은 이 환경에서 부분적으로 의미 있지만, signed container나 변경 관리 시스템이 없는 경우에만 AIS가 유의미한 gap을 메운다.

### 인증된 추론 세션

**주장**: 각 추론 요청이 특정 인증된 세션에 귀속된다.

**실제 평가**:

현재 구현에서 세션 생성은 인증 없이 이루어진다. 누구든 `/ais/v1/sessions`를 호출해 세션을 만들 수 있다. 세션이 사용자 신원과 바인딩되지 않으면 "인증된 세션"이라는 주장은 약하다. (`threat_model_limits.md` §2, TC-05)

**결론**: 현재 MVP 구현 상태에서는 이 기능이 의미 있는 보안 가치를 제공하지 못한다. 세션 생성 자체에 IAM 인증이 연결되어야 한다.

### 재전송 저항

**주장**: 동일 요청의 재전송을 탐지하고 거부한다.

**실제 평가**:

병원 LLM 환경에서 재전송 공격의 현실적 위험은 낮다. 공격자가 "진단 요청을 한 번 더 보내는" 것 자체는 큰 의미가 없다 — 같은 쿼리를 다시 보내도 결과는 유사하게 나온다. 재전송이 실질적인 피해를 만들려면 비멱등적(non-idempotent) 부작용이 있어야 하는데, 이 시나리오의 LLM은 읽기 전용 추론이다.

TLS 자체도 레코드 레이어에서 재전송을 방어한다 (`tls_overlap.md` §2.1).

**결론**: 이 시나리오에서 재전송 방어의 실질적 가치는 낮다.

### 감사 체인

**주장**: 추론별 해시 체인 감사 로그로 HIPAA 감사를 지원한다.

**실제 평가**:

이것이 이 시나리오에서 AIS의 가장 강한 주장이다.

HIPAA는 PHI(Protected Health Information) 접근에 대한 감사 로그를 요구한다. 기존 EMR 시스템은 직접 접근(의사가 환자 기록 열람)을 기록하지만, **LLM을 통한 간접 접근**은 기록하지 않을 수 있다. 즉:

- 의사가 LLM에게 "환자 X의 진단 기록을 요약해줘"라고 질문
- EMR은 LLM이 내부적으로 데이터를 읽었다는 것을 기록하지 않을 수 있음
- AIS 감사 체인은 이 추론 요청의 해시와 타임스탬프를 기록

**그러나:**
- AIS 감사는 현재 인메모리다 — 프로세스 재시작 시 소실 (`threat_model_limits.md` §2, TC-06)
- AIS 감사는 "어떤 환자 데이터가 포함되었는가"를 기록하지 않고, 요청 본문의 해시만 기록한다. HIPAA 감사관이 실제로 요구하는 것은 "어떤 환자 정보에 접근했는가"다
- SIEM 시스템 (Splunk, Elastic 등)이 이미 도입된 병원이라면 애플리케이션 레이어 로그를 더 강력하게 수집한다

**결론**: 감사 체인 아이디어는 이 환경에서 부분적으로 유용하지만, 현재 MVP 구현(인메모리, 내용 미기록)은 HIPAA 요구사항을 충족하기에 불충분하다.

### Capability 모델

**주장**: LLM이 접근할 수 있는 도구와 데이터를 세분화한다.

**실제 평가**:

AIS-Capability는 MVP에 포함되지 않는다. 설계는 존재하지만 구현되지 않았다. 현재 상태에서 이 항목은 평가할 수 없다.

## 더 강한 대안

**TEE / 원격 증명 (Intel TDX, AMD SEV):**

모델 서버를 TEE 환경에서 실행하면 소프트웨어 수준 AIS Certificate보다 훨씬 강한 모델 무결성 보증을 제공한다. 그러나 비용이 높고 운영 복잡도가 증가한다. 소규모 병원 IT 팀이 TEE를 운영하는 것은 비현실적일 수 있다.

**서명된 컨테이너 (Docker Content Trust, cosign/sigstore):**

모델 서빙 컨테이너에 서명하면 AIS AI Certificate와 유사한 모델 무결성 보증을 인프라 레이어에서 제공한다. 이미 성숙한 도구 생태계가 존재한다.

**SIEM + 애플리케이션 로깅:**

Splunk, Elastic Security 등의 SIEM 시스템이 도입된 병원에서는 LLM 애플리케이션 로그를 SIEM에 수집하는 것이 AIS 감사 체인보다 더 강력한 감사를 제공한다.

**기존 EMR 감사 확장:**

Epic이나 Cerner의 감사 기능을 LLM API 게이트웨이 레이어까지 확장하는 것이 AIS 도입보다 현실적일 수 있다.

---

## 채택 장벽

**내부 CA 운영:**

AIS AI Certificate가 의미 있으려면 병원이 내부 CA를 운영해야 한다. 이미 내부 PKI를 운영하는 병원이라면 가능하다. 그렇지 않다면 새로운 인프라 구축이 필요하다.

**운영 부담:**

AI Certificate 발행, 갱신, 폐기 프로세스를 IT 팀이 관리해야 한다. 제한된 인력을 가진 병원 IT 팀에게 이는 추가적인 운영 부담이다.

**AIS 생태계 부재:**

현재 어떤 LLM 런타임(Ollama, vLLM 등)도 AIS 프로토콜을 기본 지원하지 않는다. AIS 프록시를 직접 구축·유지해야 한다.

**현재 MVP의 감사 한계:**

현재 AIS 감사가 인메모리라면, HIPAA 감사 요구사항을 충족하기 위해서는 지속적 저장소(persistent storage)와 외부 증인(witness)이 추가로 필요하다.

## 정직한 결론

**판정: AIS 부분적으로 유용할 가능성 있음 — 단, 중요한 조건부**

AIS가 이 시나리오에서 기여할 수 있는 영역:
- 모델 무결성 바인딩 (기존 signed container 없는 경우)
- LLM 추론 요청의 해시 기반 감사 기록 (EMR이 다루지 않는 갭)

AIS가 이 시나리오에서 기여하지 못하는 영역:
- 전송 보안 (TLS가 해결)
- 사용자 인증 및 RBAC (IAM이 해결)
- 환자 기록 접근 감사 (EMR이 해결)
- 재전송 방어 (이 환경에서 위협 수준 낮음)

**핵심 조건**: AIS 감사 체인이 현재 인메모리 구현을 넘어 지속성을 갖추고, 세션 생성이 IAM과 연동되고, "환자 데이터에 대한 접근"이 감사에 명시적으로 기록되어야 한다. 이 조건들이 충족되지 않으면 AIS의 가치는 더 낮아진다.

더 강한 알터너티브가 존재하지만, TEE나 SIEM이 이미 없는 조직에서 AIS는 부분적인 보완재로 기능할 *수도* 있다. 확실하지 않다.

---

## 신뢰 수준

**신뢰도: 중간 (Medium)**

- 이 분석은 실제 병원 IT 환경 운영 경험 없이 작성되었다
- HIPAA 컴플라이언스 요구사항의 세부 해석에 따라 감사 체인의 가치가 달라질 수 있다
- AIS-Capability가 구현된다면 평가가 달라질 수 있다
- 실제 병원 IT 담당자와의 인터뷰가 이 분석의 정확도를 크게 높일 것이다
