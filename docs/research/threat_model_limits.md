# AIS 위협 모델 한계 분석

> 회의적 연구 노트 — AIS가 실제로 방어하는 위협과 방어하지 못하는 위협

---

## 1. 핵심 질문

AIS 명세서는 6개의 위협(TC-01~TC-06)을 정의하고 이에 대한 대응 메커니즘을 제시한다. 이 분석은 다음을 평가한다:

1. 각 위협에 대한 AIS의 방어가 실제로 유효한가?
2. AIS 위협 모델이 다루지 않는 현실적 위협은 무엇인가?
3. AIS의 보안 보증이 성립하기 위한 전제 조건들은 달성 가능한가?

---

## 2. 명세된 위협별 평가

### TC-01: 모델 가중치 변조

**AIS의 대응**: SHA3-256 모델 해시를 AI Certificate에 바인딩

**실제 방어 범위**:
- ✅ 서버 측에서 모델 파일이 교체된 경우 탐지 가능
- ✅ 전송 중 모델 파일 변조 탐지 가능 (이론상)

**한계**:
- ❌ **학습 단계 공격(Training-time attack)**: AIS는 명시적으로 "학습 단계 보안"을 Non-Goal로 선언한다. 백도어가 심어진 모델이 올바른 인증서와 함께 배포되면 AIS는 이를 탐지할 수 없다. AIS가 보증하는 것은 "서명된 가중치와 동일한 파일이 실행 중이다"이지, "그 가중치가 안전하다"가 아니다.
- ❌ **양자화/컴파일 변형**: GGUF 양자화, CUDA 커널 최적화 등으로 가중치가 변환되면 원본 해시가 달라진다. 파생 모델에 대한 인증서 체계가 없다.
- ⚠️ **검증 주체 문제**: 현재 MVP에서 모델 해시 검증은 AIS 프록시가 수행한다. 프록시 자체가 손상되면 이 검증은 무의미하다.

**결론**: TC-01에 대한 AIS 방어는 소프트웨어 수준 근사치다. 학습 단계 무결성과 운영 환경 무결성은 별도 메커니즘(TEE, 공급망 보안)이 필요하다.

---

### TC-02: 공급망 공격 (Rogue Model)

**AIS의 대응**: AI Certificate Chain — 발행자 서명 검증

**실제 방어 범위**:
- ✅ 서명되지 않은 모델 파일 배포 차단 가능
- ✅ 신뢰하지 않는 발행자의 인증서 거부 가능

**한계**:
- ❌ **CA 자체 손상**: X.509 PKI 역사에서 CA 손상 사례는 반복적으로 발생했다 (DigiNotar 2011, Comodo 2011 등). AIS의 AI Certificate CA가 손상되면 공격자는 임의 모델에 대한 유효한 인증서를 발급할 수 있다. AIS MVP에는 투명성 로그나 CA 감사 메커니즘이 없다.
- ❌ **발행자 자체가 악의적인 경우**: 모델 공급자가 의도적으로 백도어 모델을 인증서와 함께 배포하는 시나리오. AIS 위협 모델은 발행자를 신뢰한다고 가정하지만, 이것이 공급망 공격의 핵심 시나리오다.
- ❌ **인증서 폐기 없음**: 손상된 모델의 인증서를 폐기할 수 없다. 일단 발행된 인증서는 만료일까지 유효하다.

**결론**: TC-02 방어는 CA 생태계의 신뢰성에 완전히 의존한다. 신뢰 체인이 없는 현재 MVP에서는 이론적 방어에 불과하다.

---

### TC-03: 중간자 공격 (MITM)

**AIS의 대응**: Authenticated Handshake + 세션 무결성 MAC

**실제 방어 범위**:
- ✅ AIS 세션이 TLS 위에서 동작하면 이중 레이어 보호 제공
- ✅ 세션 MAC이 전송 중 변조를 탐지

**한계**:
- ❌ **TLS 없이 AIS만 사용하는 경우**: AIS 자체는 암호화를 제공하지 않는다. 현재 AIS 프록시 구현은 HTTP(비암호화) 위에서 동작하며, 세션 MAC은 무결성만 보호하고 기밀성은 보호하지 않는다.
- ❌ **MVP에서 ECDHE 핸드셰이크 미구현**: 명세서는 ECDHE 키 합의를 명시하지만, 현재 구현은 임의 생성 키를 API 응답으로 반환한다. 이는 채널이 암호화되어 있지 않다면 키 자체가 노출된다.

**결론**: TC-03 방어는 사실상 TLS에 위임된 상태다. AIS 단독으로는 MITM을 방어하지 못한다.

---

### TC-04: 재전송 공격 (Replay)

**AIS의 대응**: 단조 증가 시퀀스 카운터 + 논스

**실제 방어 범위**:
- ✅ 동일 세션 내 재전송을 탐지하고 거부
- ✅ 시퀀스 불일치 시 즉시 거부 (fail-closed)
- ✅ 벤치마크에서 확인된 효율적인 거부 (~31M ops/s)

**한계**:
- ⚠️ **세션 경계 재전송**: 세션이 종료된 후 동일한 요청을 새 세션에서 재전송하는 경우. 시퀀스 카운터는 세션별로 초기화되므로 이를 탐지할 수 없다.
- ⚠️ **타임스탬프 검증 없음**: 현재 구현에서 타임스탬프 필드는 기록되지만 검증되지 않는다. 먼 과거 타임스탬프를 가진 재전송이 시퀀스 번호만 일치하면 허용된다.

**결론**: TC-04는 AIS가 실제로 효과적으로 방어하는 위협이다. 그러나 세션 범위로 한정된다.

---

### TC-05: 세션 하이재킹 (Session Hijacking)

**AIS의 대응**: Session Key Binding

**실제 방어 범위**:
- ✅ 유효한 세션 키 없이는 올바른 MAC을 생성할 수 없음

**한계**:
- ❌ **세션 생성 엔드포인트 인증 없음**: 현재 MVP에서 `/ais/v1/sessions` 엔드포인트는 인증 없이 세션을 생성한다. 누구나 세션을 생성할 수 있으며, 세션 생성 자체가 권한 확인 없이 이루어진다.
- ❌ **세션 키 자체의 전송 보안**: 세션 생성 시 `integrity_key`가 HTTP 응답으로 반환된다. TLS 없이 이 응답이 전달되면 키가 노출된다.

**결론**: TC-05 방어는 세션 키 배포의 보안에 완전히 의존하며, 이는 현재 MVP에서 해결되지 않은 문제다.

---

### TC-06: 감사 로그 변조 (Audit Log Tampering)

**AIS의 대응**: Hash-chained Audit Log

**실제 방어 범위**:
- ✅ 사후에 로그 항목을 변조하면 체인 해시가 깨짐
- ✅ 변조 탐지 가능 (체인 검증으로)

**한계**:
- ❌ **로그 삭제**: 해시 체인은 기존 항목 수정을 탐지하지만, 로그 파일 자체를 삭제하거나 빈 체인으로 교체하는 것을 막지 못한다. 외부 증인(witness) 또는 투명성 로그 없이는 "이 감사 로그가 완전하다"는 것을 증명할 수 없다.
- ❌ **인메모리 한계**: 현재 구현은 인메모리 체인이다. 프로세스 재시작 시 로그가 소실된다.
- ❌ **단일 프로세스 보증**: 프록시가 손상되면 공격자는 임의의 체인을 구성할 수 있다. 해시 체인은 외부 감사자가 있을 때만 의미 있다.

**결론**: TC-06에 대한 AIS 방어는 분산 투명성 로그 없이는 내부 감사에만 유용하다.

---

## 3. 위협 모델이 다루지 않는 현실적 위협들

이것들은 AIS가 명시적으로 Non-Goal로 선언했거나, 위협 모델에 포함되지 않은 위협들이다. 그러나 이 위협들이 현실에서 얼마나 빈번하게 발생하는지를 고려하면, AIS가 다루는 영역이 얼마나 제한적인지 명확해진다.

### 3.1 프롬프트 인젝션 (Prompt Injection)

OWASP LLM Top 10 2025의 1위 위협. AIS는 이를 명시적으로 비목표(Non-Goal)로 선언한다. 그러나 현실 AI 시스템에서 가장 자주 악용되는 공격 벡터다.

AIS-Semantic이 이를 다루려 하지만, 설계에서 명확히 "차단만 가능, 통과 불가"로 제한하고 있어 우회 가능성이 구조적으로 존재한다.

### 3.2 학습 단계 데이터 포이즈닝 (Training Data Poisoning)

모델에 악의적 행동을 주입하는 가장 근본적인 공격. AIS는 이를 전혀 다루지 않는다. 이미 백도어가 심어진 모델에 올바른 AI Certificate를 발급하면, AIS는 이 모델을 신뢰한다.

### 3.3 서비스 거부 (DoS/DDoS)

AIS 프록시 자체가 DoS 공격 대상이 될 수 있다. 명세서에 레이트 리미팅, 서킷 브레이커 등에 대한 언급이 없다.

### 3.4 사이드 채널 공격 (Side-Channel)

추론 요청의 타이밍 정보, 에너지 소비 등을 통해 프롬프트나 모델 내부 상태를 추론하는 공격. AIS는 이를 다루지 않는다.

### 3.5 에이전트 권한 상승 (Agent Privilege Escalation)

AIS-Capability가 이 문제를 다루려 하지만, 에이전트가 허용된 도구를 사용하면서도 의도치 않은 부작용을 일으키는 시나리오(예: 허용된 파일 읽기 도구로 민감한 파일에 접근)는 Capability 모델만으로 완전히 방어할 수 없다.

## 4. AIS 위협 모델의 근본적 한계

### 4.1 인프라 신뢰 가정

AIS 위협 모델은 다음을 암묵적으로 가정한다:
- CA가 신뢰 가능하다
- 개인키가 안전하게 보관된다
- 프록시 자체가 손상되지 않았다
- 발행자가 악의적이지 않다

이 가정들이 성립하지 않으면 AIS의 모든 보안 보증이 무너진다. 그리고 이 가정들은 AIS가 해결하려는 것과 동일한 수준의 신뢰를 이미 요구한다.

### 4.2 모델 내부 블랙박스 문제

AIS는 모델 외부의 인프라 신뢰 문제만 다룬다. 모델이 어떤 출력을 생성하는지 — 즉 모델의 행동 자체 — 는 완전히 블랙박스로 남는다. 이는 AIS가 의도적으로 설정한 범위이지만, 동시에 AI 시스템에서 가장 중요한 보안 문제 중 하나가 AIS의 방어 범위 밖임을 의미한다.

### 4.3 공격자 모델의 불완전성

AIS 위협 모델은 네트워크 경로상의 공격자를 주요 위협으로 설정한다. 그러나 현실 AI 시스템에서 가장 빈번한 공격은 다음이다:

- 내부자(insider) 위협
- 공급망 타협
- 애플리케이션 레이어 취약점 (프롬프트 인젝션 등)
- 모델 자체의 취약한 동작

이 공격들 중 AIS가 효과적으로 방어하는 것은 제한적이다.

## 5. 종합 평가

| 위협 | AIS 방어 실효성 | 주요 한계 |
|---|---|---|
| TC-01 모델 변조 | 중간 | 학습 단계 공격, 파생 모델 미처리 |
| TC-02 공급망 공격 | 낮음 | CA 신뢰 체계 부재, 폐기 없음 |
| TC-03 MITM | TLS 의존 | AIS 단독으로는 미방어 |
| TC-04 재전송 | 높음 | 세션 범위 내로 한정 |
| TC-05 세션 하이재킹 | 중간 | 세션 생성 인증 부재 |
| TC-06 감사 변조 | 낮음~중간 | 외부 투명성 로그 없이 불완전 |
| 프롬프트 인젝션 | 비목표 | 현실에서 가장 빈번한 공격 |
| 학습 데이터 포이즈닝 | 비목표 | 모델 내부 신뢰 문제 미해결 |

---

## 6. 결론: 위협 모델이 제안하는 올바른 방향

AIS 위협 모델의 한계를 비판하는 것은 쉽지만, AIS가 시도하는 방향 자체는 의미 있다. AI 인프라에 대한 결정론적 신뢰 보증을 추구하는 것은 올바른 연구 방향이다.

그러나 현재 위협 모델은 다음을 명확히 해야 한다:

1. **"AIS는 TLS + mTLS + 감사 시스템의 대체재가 아니라 보완재다."** — 이를 더 명확히 명시해야 한다.

2. **"현재 MVP의 보안 보증 범위는 단일 조직 내 폐쇄 환경으로 제한된다."** — 개방 생태계에서의 주장은 투명성 로그, 폐기 인프라, CA 생태계가 갖추어진 후에야 가능하다.

3. **위협 TC-04(재전송 방어)가 AIS MVP에서 가장 검증된 보안 기능이다.** — 이 영역을 핵심 가치 제안으로 명확히 하고, 나머지는 "미래 연구"로 솔직하게 표현하는 것이 바람직하다.

---

*이 문서는 AIS 프로젝트의 연구 정직성을 높이기 위해 작성되었다. 위협 모델의 한계를 솔직하게 인정하는 것이 보안 연구의 신뢰성을 높이는 데 기여한다.*
