# AIS와 TLS의 중복 분석

> 회의적 연구 노트 — AIS가 TLS 위에 실제로 무엇을 추가하는가?

---

## 1. 핵심 질문

AIS 명세서는 스스로를 "AI 시스템의 TLS"로 정의한다. 그러나 현대적 TLS 배포(TLS 1.3 + mTLS + X.509)가 이미 다음을 제공한다:

- 전송 기밀성
- 서버/클라이언트 상호 인증
- 재전송 공격 방지 (record sequence number)
- 무결성 보호 (AEAD)
- 인증서 기반 신원 확인

그렇다면 AIS가 실제로 추가하는 것은 무엇인가?

---

## 2. TLS가 이미 해결하는 문제들

### 2.1 전송 레이어 보안

현재 OpenAI API, Ollama, vLLM 등 모든 주요 AI 백엔드는 HTTPS(TLS 위의 HTTP)를 통해 통신한다. 즉:

- 클라이언트와 서버 사이의 패킷 관찰·수정 → TLS AEAD로 방어됨
- 중간자 공격(MITM) → 서버 인증서로 방어됨
- 재전송 공격 → TLS 레코드 시퀀스 번호로 방어됨

AIS의 위협 모델(TC-03, TC-04)에서 명시된 MITM 및 재전송 공격은 **TLS가 이미 다루는 영역**이다.

### 2.2 세션 무결성

TLS 세션은 자체적으로:

- 핸드셰이크 기반 세션 키 확립
- 레코드 단위 MAC 검증
- 세션 재개(session resumption) 메커니즘

을 포함한다. AIS의 세션 MAC(SHA3-256 기반 integrity_mac)은 TLS의 AEAD 태그와 기능적으로 유사하며, TLS 위에서 이를 중복 적용하는 셈이다.

### 2.3 상호 인증 (mTLS)

mTLS(Mutual TLS)는 클라이언트와 서버 양방향 X.509 인증서를 요구한다. 이미 서비스 메시(Istio, Envoy), API 게이트웨이, 클라우드 인프라에서 표준적으로 사용된다. AI 백엔드 접근 제어에도 동일하게 적용 가능하다.

---

## 3. AIS가 실제로 추가하는 영역

TLS와의 차별점이 있는 영역은 다음으로 한정된다.

### 3.1 모델 가중치 무결성 바인딩

**TLS가 하지 않는 것**: 서버 인증서는 서버 *프로세스*를 인증하지, 서버가 실행하는 *모델 파일*을 인증하지 않는다.

AIS의 AI Certificate는 다음을 바인딩한다:

```
모델 파일 해시 (SHA3-256)
    ↓
발행자 서명 (Ed25519)
    ↓
인증서
```

즉, 서버가 올바른 TLS 인증서를 가지고 있어도, 내부에서 변조된 모델을 실행하고 있다면 TLS는 이를 탐지할 수 없다. AIS의 AI Certificate는 이 특정 갭을 메운다.

**반론**: 이 문제는 서버 측 코드 서명(Secure Boot, TPM 기반 원격 증명, Docker Image Digest)으로도 해결 가능하다. Intel TDX, AMD SEV 같은 TEE 환경에서는 원격 증명이 이 역할을 더 강력하게 수행한다. AIS의 AI Certificate는 TEE 없이 동작하는 소프트웨어 수준 근사치다.

### 3.2 애플리케이션 레이어 감사 체인

TLS는 전송 레이어 보안만 다루며, 애플리케이션 레이어에서 무슨 요청이 이루어졌는지 기록하지 않는다.

AIS의 해시 체인 감사 로그는:

- 각 추론 요청의 해시를 기록
- 이전 엔트리와 연결 (블록체인 유사 구조)
- 사후 감사(post-hoc audit)를 위한 변조 증거 보존

를 제공한다. 이는 TLS 레벨에서는 원칙적으로 불가능한 애플리케이션 레이어 감사다.

**반론**: 동일한 감사 기능은 일반적인 로그 서명 시스템(예: Google's Trillian, Apache Kafka + 서명)으로 구현 가능하다. AIS가 제공하는 것은 AI 추론에 특화된 편의 래퍼(wrapper)에 가깝다.

### 3.3 모델별 능력 경계 (AIS-Capability)

OAuth 2.0 스코프나 JWT 클레임과 유사하지만, AI 에이전트의 도구 호출(tool call)에 특화된 권한 모델이다. TLS 자체에는 이런 애플리케이션 수준 권한 모델이 없다.

**반론**: 기존 OAuth 2.0 + JWT로 충분히 표현 가능하며, 이미 대부분의 API 게이트웨이가 이를 구현한다.

---

## 4. TLS와의 중복 요약

| AIS 기능 | TLS 1.3 + mTLS로 대체 가능? | 비고 |
|---|---|---|
| 전송 암호화 | ✅ 완전 대체 가능 | TLS AEAD가 더 강력 |
| MITM 방어 | ✅ 완전 대체 가능 | TLS 인증서 핀닝 포함 |
| 재전송 방어 | ✅ 대체 가능 | TLS 레코드 시퀀스로 충분 |
| 세션 무결성 MAC | ✅ 대체 가능 | TLS AEAD로 중복 |
| 클라이언트 인증 | ✅ mTLS로 대체 가능 | |
| **모델 파일 해시 바인딩** | ❌ TLS 범위 밖 | AIS만의 기여 |
| **추론별 감사 체인** | ⚠️ 부분 대체 가능 | 일반 감사 시스템으로 가능하나 AI 특화 아님 |
| **에이전트 Capability 토큰** | ⚠️ OAuth로 대체 가능 | AI 에이전트 특화 아님 |
| 프록시 투명성 | ⚠️ 아키텍처 의존 | 기존 API 게이트웨이로 가능 |

## 5. 결론: AIS는 필요한가?

### 강한 반론

1. **대부분의 환경은 TLS만으로 충분하다.** 클라우드 제공 AI API(OpenAI, Anthropic, Google)를 사용하는 경우, 모델 파일 무결성은 공급자 책임이며 클라이언트가 검증할 필요가 없다.

2. **로컬 모델 환경에서도 대안이 있다.** Docker Image Digest, Secure Boot, TEE 원격 증명이 AIS AI Certificate보다 더 강한 보증을 제공한다.

3. **AIS 자체도 TLS 위에서 동작해야 한다.** AIS 프록시가 HTTP 위에서 동작한다면, AIS 세션 MAC은 TLS가 이미 제공하는 것을 중복으로 구현하는 것이다.

### 합리적인 사용 사례

다음 조건이 모두 충족될 때 AIS의 부가 가치가 명확해진다:

1. **로컬/자체 호스팅 AI 인프라**를 운영하는 환경
2. **TEE 없이** 소프트웨어 수준 모델 무결성 보증이 필요한 경우
3. **AI 추론별 감사 로그**가 규제 또는 내부 정책으로 요구되는 경우
4. **AI 에이전트 도구 호출**에 대한 세분화된 권한 제어가 필요한 경우

이 조건들 중 일부만 해당되는 경우, 기존 인프라(mTLS + 감사 시스템 + OAuth)로 충분할 가능성이 높다.

---

## 6. 연구 방향 제안

AIS가 TLS와의 명확한 차별성을 확보하려면 다음이 필요하다:

1. **TEE 연동**: 소프트웨어 수준 AI Certificate는 TEE 원격 증명과 결합될 때만 강한 보증을 제공한다.
2. **투명성 로그**: AI Certificate 발행을 외부 투명성 로그에 기록하지 않으면 악의적 발행자가 위조 인증서를 발급할 수 있다.
3. **TLS 위의 AIS**: AIS는 TLS를 대체하려 해서는 안 되며, TLS 위에서 애플리케이션 레이어의 AI 특화 보증을 추가하는 역할로 명확히 위치 지어야 한다.
